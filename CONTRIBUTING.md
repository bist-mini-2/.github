# 🤝 기여 가이드라인 (Contributing Guidelines)

우리 팀의 일원으로서 프로젝트에 기여해 주셔서 감사합니다! 이 문서는 원활한 협업과 코드 품질 유지를 위해 팀 전체가 준수해야 하는 개발 컨벤션 및 기여 프로세스를 안내합니다.

---

## 🌿 브랜치 전략 (Branching Strategy)

우리 팀은 **GitHub Flow** 기반의 브랜치 전략을 사용합니다. 모든 작업은 개별 브랜치에서 진행되며, Pull Request를 통해 코드 리뷰를 거친 후 메인 브랜치에 병합됩니다.

### 브랜치 유형 및 네이밍 규칙
브랜치 이름은 `작업유형/기능요약` 형태로 소문자와 하이픈(`-`)을 사용하여 작성합니다.

- **`main`**: 프로덕션 배포 브랜치입니다. 언제나 즉시 배포할 수 있는 상태로 유지되어야 합니다.
- **`feature/`**: 새로운 기능 개발
  - 예시: `feature/login-form`, `feature/payment-api`
- **`fix/`**: 버그 수정
  - 예시: `fix/token-expiration`, `fix/ui-overflow`
- **`refactor/`**: 코드 리팩터링
  - 예시: `refactor/clean-auth-logic`
- **`docs/`**: 문서 생성 및 수정
  - 예시: `docs/api-specification`
- **`design/`**: CSS, 퍼블리싱 등 UI 디자인 스타일 작업
  - 예시: `design/landing-page-hero`

### 작업 및 배포 흐름
1. `main` 브랜치에서 최신 코드를 pull 받습니다.
2. 새로운 작업 브랜치를 생성합니다 (`git checkout -b feature/your-feature`).
3. 변경 사항을 구현하고 커밋을 작성합니다.
4. 원격 저장소(Origin)에 브랜치를 푸시합니다.
5. GitHub에서 `main` 브랜치를 대상으로 Pull Request (PR)를 생성합니다.
6. 팀원의 코드 리뷰와 피드백을 수용하여 보완합니다.
7. 최소 1명 이상의 **Approve**와 자동화 빌드/테스트 성공 확인 후 머지(Merge)합니다.

## 💬 커밋 메시지 규칙 (Commit Message Conventions)

일관된 커밋 히스토리를 위해 **Conventional Commits** 양식을 준수합니다.

### 커밋 메시지 구조
```
<type>(<scope>): <subject>  # 제목 (최대 50자, 영어/한글 모두 가능하나 명확하게)

<body>                      # 본문 (생략 가능, 구체적인 변경 사항 기술)

<footer>                    # 바닥글 (생략 가능, 연관된 Issue 번호 표기)
```

### 커밋 타입 (Type)
| 타입 | 설명 |
| :--- | :--- |
| **`feat`** | 새로운 기능 추가 |
| **`fix`** | 버그 수정 |
| **`design`** | CSS 및 UI 스타일 변경 작업 |
| **`refactor`** | 코드 리팩터링 (기능 추가나 버그 수정이 없는 변경) |
| **`docs`** | 문서 변경 (README.md, API 문서 등) |
| **`style`** | 코드 포맷팅, 세미콜론 누락 등 (코드 자체 동작의 변화가 없는 경우) |
| **`test`** | 테스트 코드 추가 또는 수정 |
| **`chore`** | 빌드 업무 수정, 패키지 매니저 설정, 프로젝트 설정 변경 등 |
| **`rename`** | 파일 혹은 폴더명 변경, 위치 이동 |
| **`remove`** | 파일 삭제 |

### 작성 규칙
- 제목 첫 글자는 대문자를 지양하고, 마침표(`.`)는 붙이지 않습니다.
- 제목은 동사 원형(영어 기준)이나 명확한 개조식(한글 기준)으로 작성합니다.
- 본문은 **어떻게(How)**보다 **무엇을(What)**, **왜(Why)** 변경했는지 작성합니다.

### 커밋 메시지 예시
```text
feat(auth): 카카오 소셜 로그인 구현

- 카카오 SDK 연동 및 토큰 발급 프로세스 추가
- 사용자 프로필 정보 데이터베이스 매핑 로직 구현

Closes #42
```

## 🔍 코드 리뷰 및 PR 프로세스

- **PR 제목**: 커밋 컨벤션과 동일하게 제목을 작성합니다. (예: `feat(auth): 카카오 로그인 연동`)
- **템플릿 작성**: PR 작성 시 제공되는 템플릿의 항목들을 성실히 기재합니다.
- **Self Review**: PR을 생성하기 전 본인의 변경 사항을 다시 한번 검토합니다. 불필요한 주석, 디버깅용 로그(`console.log`, `print` 등)가 남아있지 않은지 확인합니다.
- **Reviewer 지정**: 관련 도메인을 가장 잘 아는 팀원 또는 협업자를 리뷰어로 지정합니다.
